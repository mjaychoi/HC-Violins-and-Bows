# 사용자 경험 문제점 및 개선 사항

이 문서는 유저 관점에서 발견된 아쉬운 점과 이상한 점들을 정리합니다.

## 🔴 심각한 문제 (Critical Issues)

### 1. 네비게이션 구조 혼란

**문제점**:
- 사이드바에는 6개 메뉴만 표시되는데, 실제로는 더 많은 페이지가 존재
- **사이드바 메뉴**: Items, Clients, Connected Clients, Calendar, Client Analytics, Sales
- **실제 존재하는 페이지**: `/dashboard`, `/clients`, `/connections`, `/calendar`, `/clients/analytics`, `/sales`, **`/instruments`**
- `/instruments` 페이지가 사이드바에 없어서 접근 불가능 (직접 URL 입력 필요)
- 사용자가 이 페이지의 존재를 알 수 없음
- `/customer`는 deprecated되어 `/clients/analytics`로 리다이렉트됨 ✅

**영향**:
- 숨겨진 기능 발견 불가
- 직관적이지 않은 UX
- 일관성 없는 정보 아키텍처

**개선 제안**:
- 모든 주요 페이지를 사이드바에 포함하거나
- 페이지 통합 (예: `/dashboard`와 `/instruments` 통합, `/customer`와 `/clients` 통합)

---

### 2. URL과 라벨 불일치

**문제점**:
- ~~`/form` → "Connected Clients"~~ ✅ **해결됨**: `/connections`로 변경됨
- ~~`/dashboard` → 사이드바에는 "Items"~~ ✅ **해결됨**: Dashboard 페이지 제목을 "Items"로 변경하여 사이드바 라벨과 일치
- ~~`/customer` vs `/clients`~~ ✅ **해결됨**: `/customer`는 deprecated되어 `/clients/analytics`로 리다이렉트됨

**영향**:
- URL만 봐서는 페이지 기능을 알 수 없음 (개선됨)
- 북마크나 공유 시 혼란 (개선됨)
- SEO 및 접근성 문제 (개선됨)

**개선 상태**:
- ✅ `/form` → `/connections`로 변경 완료
- 명확한 URL 구조: `/instruments`, `/clients`, `/connections` 등

---

### 3. 기능 중복 및 혼란

**문제점**:

#### 3.1 Dashboard vs Instruments
- `/dashboard` (Items): 악기 목록, 추가/수정/삭제
- `/instruments`: 악기 목록, 클라이언트 연결
- 두 페이지가 유사한 기능을 제공하지만 사이드바에는 Dashboard만 표시

**질문**: Instruments 페이지의 목적이 무엇인가?

#### 3.2 Customer vs Clients vs Client Analytics
- ~~`/customer`~~: **Deprecated** - `/clients/analytics`로 리다이렉트됨 ✅
- `/clients`: 클라이언트 관리 (추가/수정/삭제)
- `/clients/analytics`: 클라이언트별 구매 이력 및 통계

**상태**: Customer 페이지는 이미 deprecated 처리되어 Client Analytics로 통합됨. 하지만 URL이 아직 존재함.

**영향**:
- 사용자가 어느 페이지를 사용해야 할지 혼란
- 중복된 기능으로 인한 데이터 불일치 가능성
- 학습 곡선 증가

**개선 제안**:
- 페이지 통합 또는 명확한 차별화
- 각 페이지의 목적을 명확히 정의하고 사용자 가이드 제공

## 🟠 중간 수준 문제 (Medium Issues)

### 4. 로그인 페이지의 "LATEST" 표시

**문제점**:
```typescript
// src/app/page.tsx:73
<h2 className="mt-6 text-center text-3xl font-extrabold text-gray-900">
  Instrument Inventory App - LATEST
</h2>
```

**영향**:
- 프로덕션 환경에서 "LATEST" 같은 개발용 라벨이 표시됨
- 전문성이 떨어져 보임
- 사용자에게 불필요한 정보

**개선 제안**:
- "LATEST" 제거 또는 환경 변수로 관리
- 프로덕션에서는 간단히 "Instrument Inventory App"만 표시

---

### 5. 빈 상태(Empty State) 처리 일관성 부족

**문제점**:
- 일부 페이지는 상세한 Empty State (아이콘, 설명, 액션 버튼)
- 일부 페이지는 단순 텍스트만 ("No data found")
- 일부 페이지는 Empty State가 없을 수 있음

**예시**:
- ✅ Instruments 페이지: 아이콘 + 설명 + "Add Instrument" 버튼
- ⚠️ Sales 페이지: "No sales found for the selected filters." (단순 텍스트)

**영향**:
- 일관성 없는 UX
- 사용자가 다음 액션을 취하기 어려움

**개선 제안**:
- 모든 페이지에 일관된 Empty State 컴포넌트 사용
- 적절한 아이콘, 설명, 다음 액션 안내 포함

---

### 6. 로딩 상태 일관성 부족

**문제점**:
- 일부 페이지: Skeleton UI (Dashboard ItemList)
- 일부 페이지: 단순 텍스트 "Loading..." (Clients dynamic import)
- 일부 페이지: Spinner

**예시**:
```typescript
// Dashboard: Skeleton 사용
loading: () => <TableSkeleton rows={8} columns={7} />

// Clients: 단순 텍스트
loading: () => <div>Loading list...</div>
```

**영향**:
- 로딩 경험의 일관성 부족
- 사용자가 페이지가 로딩 중인지 확실하지 않을 수 있음

**개선 제안**:
- 모든 페이지에서 동일한 Skeleton 패턴 사용
- 페이지 타입에 맞는 적절한 Skeleton 컴포넌트 (Table, Card, List 등)

---

### 7. 확인 다이얼로그 일관성

**문제점**:
- 대부분의 페이지: 커스텀 `ConfirmDialog` 컴포넌트 사용 ✅
- 테스트 코드 확인 결과 `window.confirm()` 사용하지 않음 ✅
- 하지만 코드베이스 전체를 완전히 확인하지 못했으므로 추가 확인 필요

**상태**: 대부분 개선되었지만 전체 확인 필요

**개선 제안**:
- 코드베이스 전체에서 `confirm(` 또는 `window.confirm` 검색하여 완전히 제거 확인
- 모든 확인 다이얼로그에서 커스텀 `ConfirmDialog` 컴포넌트 사용 강제

---

### 8. 에러 메시지 표시 방식

**문제점**:
- 일부는 Toast 알림
- 일부는 인라인 에러 메시지
- 일부는 모달

**영향**:
- 사용자가 에러를 놓칠 수 있음
- 일관성 없는 피드백

**개선 제안**:
- 에러 타입에 따른 표시 방식 명확화
  - 폼 에러: 인라인 (필드 근처)
  - API 에러: Toast 알림
  - 치명적 에러: 모달

## 🟡 작은 문제 (Minor Issues)

### 9. 페이지 제목과 헤더 불일치 가능성

**문제점**:
- 사이드바 라벨: "Items"
- 실제 페이지 제목: "Dashboard" 가능성
- 헤더에 표시되는 제목 확인 필요

**영향**:
- 사용자가 현재 어느 페이지에 있는지 혼란

**개선 제안**:
- 모든 페이지에서 헤더 제목과 사이드바 라벨 일치 확인
- 명확한 페이지 제목 사용

---

### 10. 검색/필터 기능 위치 불일치

**문제점**:
- 페이지마다 검색/필터 UI 위치가 다를 수 있음
- 일부는 상단, 일부는 사이드바, 일부는 헤더

**영향**:
- 사용자가 검색 기능을 찾기 어려움
- 학습 곡선 증가

**개선 제안**:
- 모든 페이지에서 검색/필터 위치 통일 (예: 항상 상단 또는 항상 헤더 아래)

---

### 11. 모달 닫기 방식

**문제점**:
- 일부 모달: X 버튼만
- 일부 모달: X 버튼 + ESC 키
- 일부 모달: 배경 클릭으로 닫기 가능
- 일부 모달: 배경 클릭으로 닫기 불가

**영향**:
- 일관성 없는 인터랙션
- 사용자가 모달을 닫는 방법을 예측하기 어려움

**개선 제안**:
- 모든 모달에서 동일한 닫기 방식 적용
- 권장: X 버튼 + ESC 키 + 배경 클릭 (폼 입력 중일 때는 배경 클릭 방지 옵션)

---

### 12. 성공 메시지 표시

**문제점**:
- 일부 작업 성공 후: Toast 알림
- 일부 작업 성공 후: 페이지 새로고침만
- 일부 작업 성공 후: 피드백 없음

**영향**:
- 사용자가 작업이 성공했는지 확인하기 어려움
- 불안감 (저장했는지 확실하지 않음)

**개선 제안**:
- 모든 중요한 액션 (생성, 수정, 삭제) 후 Toast 성공 메시지 표시
- 일관된 메시지 형식 사용

---

### 13. 페이지 전환 시 데이터 보존

**문제점**:
- 필터 상태가 페이지 전환 시 보존되는지 불명확
- 검색어가 페이지 전환 시 유지되는지 불명확

**영향**:
- 사용자가 작업 흐름이 끊김
- 반복 작업 필요

**개선 제안**:
- URL 쿼리 파라미터로 필터/검색 상태 보존
- 또는 localStorage를 통한 상태 보존 (선택적)

---

### 14. 반응형 디자인 일관성

**문제점**:
- 모든 페이지가 모바일에서 잘 작동하는지 확인 필요
- 사이드바 접기/펼치기 동작이 모바일에서 일관적인지 확인 필요

**영향**:
- 모바일 사용자 경험 저하

**개선 제안**:
- 모든 페이지의 모바일 버전 테스트
- 사이드바는 모바일에서 기본적으로 접혀있고, 오버레이 형태로 표시
